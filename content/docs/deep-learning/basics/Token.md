# Token

Token 是大语言模型（LLM）处理自然语言时的**最小输入单元**，可以理解为模型“读懂文字”的基本颗粒度。

简单来说，Token 就是对自然语言进行分词后得到的结果。

例如句子“我想要吃苹果”，根据分词规则不同，Token 切分结果可能是：

- 粒度较粗：我、想要、吃、苹果
- 粒度较细：我、想、要、吃、苹、果

这里需要注意：**对单个固定模型来说，分词是确定性算法，不会今天这样切、明天那样切。**不同的分词结果，只来自不同模型、不同分词器的规则差异。

负责执行分词的工具就是 LLM 专用的 **Tokenizer**。它的**输入是原始文本字符串**，**输出是对应的 Token ID 序列**，整个过程只做文本切分与编号映射。

# Tokenizer

Tokenizer 不属于神经网络，没有可学习参数、不参与反向传播，也不是分类、回归或生成模型，本质是基于词频统计的规则化字符串处理算法（如 BPE），和具备理解、推理能力的深度神经网络在结构与功能上完全不同。

中英文本在 Token 切分上存在明显差异：
- 英文通常先按空格拆分单词，再在单词内部拆分为词根、词缀等子词单元；
- 中文没有天然空格分隔，Tokenizer 会直接从单字层级开始，依据词频合并为多字单元，因此中文 Token 粒度更贴近单字或常用词组。

## 中文 Tokenizer 怎么切？
Tokenizer 内部有一个预先训练好的 merges.txt，长这样：
```text
想 要 → 想要
苹 果 → 苹果
```
它会从头到尾扫描，遇到能合并的就合并：

扫描到 想 + 要 → 合并成 想要
扫描到 苹 + 果 → 合并成 苹果

最终变成：我、想要、吃、苹果

## 英文 Tokenizer 怎么切？
以最常见的句子：
```
I like eating apples
```
1. 先按空格切成单词
    I, like, eating, apples
2. 再把每个单词拆成最小单位（字符）
    I → I
    like → l, i, k, e
    eating → e, a, t, i, n, g
    apples → a, p, p, l, e, s
3. 根据训练好的合并表贪心合并

比如合并表里有：

```text
e + at → eat
eat + ing → eating
app + le → apple
apple + s → apples
```

最后就会合并成：I, like, eating, apples

# 向量化

分词完成后，每个 Token 都会被**向量化**，转换成模型可以计算的数值向量。

这个过程本质上是**查表映射**：模型内部维护着词向量表，查询到“我”对应的向量为 $X_1$，其他 Token 同理依次转换。

大语言模型本身无法直接理解自然语言文本，只能处理数值向量。Token 只是连接自然语言与向量表示的中间桥梁，只有完成切分与向量化，文本才能进入模型被真正计算。

# 上下文与计费

Token 数量直接决定了模型使用成本与上下文限制：

一方面，OpenAI 等平台会按输入+输出的总 Token 数计费，Token 越多费用越高；

另一方面，所有 LLM 都有上下文窗口上限，一旦 Token 总数超出限制，文本会被截断或无法提交，过长内容也会显著影响模型推理效率与效果。
